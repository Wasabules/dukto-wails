# Wails/Go/Svelte Port ŌĆö Scope and Coexistence Plan

Status: **shipped in this repo** under `wails/`. The desktop frontend (Windows, macOS, Linux) runs on Wails v2 + Go + Svelte-TS; the Qt6 tree at the repo root now serves Android only. The two codebases share the LAN wire format defined in `PROTOCOL.md` and must stay interoperable with each other and with third-party Dukto peers.

## Port layout (Go + Svelte)

Root of the port is `wails/` at the repo top. The Go side is organised into one `main` package that wires Wails up plus small reusable packages under `wails/internal/`:

| Package                     | Purpose |
|-----------------------------|---------|
| `internal/protocol`         | Pure-Go encoder/decoder for the Dukto wire format (UDP `BuddyMessage`, TCP `SessionHeader`/`ElementHeader`, `BuildSignature`). No Wails/Qt deps; single source of truth for byte-level I/O. |
| `internal/discovery`        | UDP messenger (HELLO/GOODBYE), self-echo suppression, per-IP HELLO cooldown, broadcast-storm guard. |
| `internal/transfer`         | TCP server + receiver pipeline; sender; `AllowSession` hook for pre-session policy gates. |
| `internal/settings`         | JSON-file-backed settings store with atomic `Update(func(*Values))`. Handles one-shot migration from the Qt `QSettings` store on first run. |
| `internal/history`          | Recent-activity list (sent/received). |
| `internal/audit`            | Append-only JSON-per-line audit log with 10 MiB rotation (`audit.log`, `audit.log.1`), 0o600 perms. |
| `internal/platform`         | OS name token (`Windows`/`Macintosh`/`Linux`), user/hostname, default download path, dark-theme detection. |
| `internal/osint`            | Small cross-platform OS shims kept out of `platform` (e.g. open-file-in-explorer). |
| `internal/httpserve`        | Avatar HTTP side-channel on `udp_port + 1`. |
| `internal/avatar`           | Avatar PNG resolution (64├Ś64, local + per-peer cache). |

Top-level `wails/` files:

- `main.go` ŌĆö Wails app bootstrap.
- `app.go` ŌĆö `App` struct holding the bound surface.
- `lifecycle.go` ŌĆö startup/shutdown wiring (messenger, server, settings, audit, event routing).
- `events.go` ŌĆö Wails event names emitted to the frontend.
- `bindings_*.go` ŌĆö Go methods bound into JS via Wails (`bindings_files.go`, `bindings_history.go`, `bindings_peers.go`, `bindings_policy.go`, `bindings_receive.go`, `bindings_security.go`, `bindings_settings.go`, `bindings_transfer.go`). Grouped by concern; the same `App` receiver is shared across them.
- `disk_unix.go` / `disk_windows.go` ŌĆö build-tagged free-space probe for the disk guard.

Frontend (`wails/frontend/`) is Svelte + Vite + TypeScript:

- `src/App.svelte` ŌĆö state orchestration and composition of the other components.
- `src/components/` ŌĆö one file per UI surface (`Header`, `PeerList`, `Composer`, `ReceivedList`, `ProgressStack`, `PreviewModal`, `PendingSessionModal`, `SettingsModal`, ŌĆ”).
- `src/components/settings/` ŌĆö one tab per file (`GeneralTab`, `SecurityTab`, `LimitsTab`, `NetworkTab`, `AuditTab`).
- `src/lib/dukto.ts` ŌĆö typed wrappers over the Wails-generated bindings; event subscription helpers live in `src/lib/events.ts`; Svelte stores under `src/lib/stores/`.
- `wailsjs/` ŌĆö generated GoŌåöJS bindings (committed; regenerated by `wails generate`).

## Feature coverage vs Qt6

### Parity (done)

- UDP discovery + TCP transfer (full implementation of `PROTOCOL.md`).
- Send/receive files, folders, text, clipboard text.
- Recent-activity list, destination folder chooser.
- Profile (buddy name, avatar, avatar HTTP side-channel on port 4645).
- System tray with close-to-tray, notifications on receive.
- Dark/light theme detection.
- Native per-OS dark-theme detection (same tokens as `platform.cpp`).
- Settings persisted in a Wails-native JSON store with first-run Qt `QSettings` migration (see ┬¦Interop with existing installs).
- Cross-subnet manual peers.

### Deferred / not ported

- Screen capture ŌĆö not yet implemented in the Wails app.
- Single-instance enforcement ŌĆö not yet implemented (tracked under ┬¦Open questions).
- Windows taskbar progress (`ITaskbarList3`) ŌĆö not yet implemented.
- Custom theme colour picker ŌĆö Wails UI uses a fixed palette; user-chosen `ThemeColor` is ignored.
- `UPDATER` / `updateschecker.cpp` ŌĆö intentionally dropped (upstream marks it broken).
- libnotify-specific notification path ŌĆö Wails uses a cross-platform notify shim; no libnotify branch.
- Legacy `qml/old/` UI variant ŌĆö Svelte is the only UI in the port.

### Beyond parity (security hardening added in the port)

These do not exist in the Qt6 app and are Wails-only for now:

- **Audit log** ŌĆö append-only JSON events under `<UserConfigDir>/dukto/audit.log` for every accept/reject/policy-hit decision. 10 MiB rotation. Viewable from the Audit settings tab.
- **Allow-list (whitelist)** ŌĆö only accept transfers from buddies approved by signature.
- **Block list** ŌĆö hard-reject by signature (runs before rate limits and whitelist).
- **Auto-reject file extensions** ŌĆö session ends cleanly if any element name matches the blocked-extension list.
- **Large-session threshold** ŌĆö reject sessions whose total size exceeds a configurable MB cap.
- **Confirm unknown peers** ŌĆö first session from a never-seen buddy pops a 60-second modal (Allow/Reject); decision is remembered until the user clicks "Forget approvals".
- **Files-per-session cap** and **path-depth cap** ŌĆö reject oversized session manifests before any bytes are read.
- **Minimum free-disk percent guard** ŌĆö reject sessions that would take the destination below a configurable percentage of total capacity.
- **TCP per-IP accept cooldown** ŌĆö drops repeated inbound connections from the same IP within a window.
- **UDP HELLO per-source-IP cooldown** ŌĆö suppresses noisy discovery floods.
- **Allowed network interfaces filter** ŌĆö the TCP server only accepts on the selected interfaces; unchecked = listen everywhere (default).
- **Idle auto-disable** ŌĆö master "accept receiving" switch flips off after N minutes of inactivity.

Defense-in-depth order on the receive path is: master switch ŌåÆ block list ŌåÆ TCP rate limit ŌåÆ interface filter ŌåÆ whitelist ŌåÆ confirm-unknown ŌåÆ extension/size/file-count/depth/disk checks on the session header.

## Scope of the Qt6 codebase after the port

Two transitions are now in flight:

1. **Wails replaces Qt for the desktop.** Once the Wails app reaches the users currently on the Qt6 desktop builds, the desktop-only pieces below can be deleted from the root tree.
2. **`android-native/` (Kotlin + Compose) replaces Qt for Android.** Once the native APK has been used in the wild for a release cycle, the entire Qt-Android tooling (the Android `qml/new/` UI, JNI helpers, the Android branches of `platform.cpp`, the Qt-Android workflows) can also be removed ŌĆö at which point this Qt repo no longer ships anything user-facing and can be archived.

### Desktop-only (delete after Wails takes over)

- `ecwin7.*` (Windows taskbar).
- `platform.cpp` branches for `Q_OS_WIN`, `Q_OS_MAC`, `Q_OS_LINUX` non-Android ŌĆö keep only the `Q_OS_ANDROID` paths until step 2 fires.
- `systemtray.*` ŌĆö desktop-only, gated by `DESKTOP_APP`.
- `modules/SingleApplication` submodule ŌĆö desktop-only.
- Linux DBus integration, macOS bundle/icon setup, Windows `.rc`/`.ico`.
- `qml/old/` ŌĆö Qt6-Android uses the `qml/new/` tree unconditionally.
- `dukto.desktop`, `.png` icon install rules, macOS `.icns`.
- CMake `USE_SINGLE_APP`, `USE_NOTIFY_LIBNOTIFY`, `USE_UPDATER` options.

### Android-only (delete after the native APK takes over)

- `qml/new/` and `qml/common/` (the entire Quick Controls UI).
- `androidutils.{h,cpp}` (JNI helpers).
- `android/qt5/` and `android/qt6/` Gradle / manifest scaffolding bundled with Qt-Android.
- `Q_OS_ANDROID` branches in `platform.cpp`.
- `network/` and `*` desktop-shared classes IF nothing else still depends on them at that point ŌĆö Wails has its own Go reimplementation, so the C++ network code's only consumer is the Qt UI.
- `dukto.pro`, `CMakeLists.txt` (in their entirety once neither desktop nor Android builds use them).
- Workflows: `build-android.yml`, `build-qt5.yml`, `build-qt6.yml`, and the `qt5` / `qt6` / `android` jobs in `release.yml`.

**Don't touch yet.** Do each cleanup in one go only after the corresponding rewrite has replaced the Qt build for real users, so bisecting regressions against the old Qt build is still possible during the transition.

## Interop constraints (non-negotiable)

1. **Wire format**: exactly as specified in `PROTOCOL.md`. Test both stacks against the same fixture bytes.
2. **UDP/TCP ports**: 4644 by default, configurable; avatar HTTP at `udp_port + 1`.
3. **Platform token in signature**: use `Windows`, `Macintosh`, `Linux` (not `"Win"`, `"macOS"`, `"Ubuntu"`, etc.) ŌĆö legacy peers render the platform icon off this string.
4. **Text-snippet magic**: literal `___DUKTO___TEXT___`.
5. **Directory sentinel**: element_size `-1`.
6. **Little-endian** integer I/O regardless of host.

If anything on this list needs to change, it is a protocol version bump and must be coordinated with every implementation (ours + third-party).

## Interop with existing installs (settings migration)

The Qt app stores settings under the vendor/application pair `msec.it` / `Dukto` via `QSettings`. On each OS:

- **Windows**: `HKCU\Software\msec.it\Dukto` (registry). Values are typed (`REG_SZ`, `REG_DWORD`, `REG_BINARY`).
- **macOS**: `~/Library/Preferences/com.msec.it.Dukto.plist` (or `it.msec.Dukto` depending on Qt version ŌĆö verify at migration time).
- **Linux**: `~/.config/msec.it/Dukto.conf` (INI-like text).

Keys currently written (`settings.cpp`):

| Key                    | Type      | Default             | Notes |
|------------------------|-----------|---------------------|-------|
| `DestPath`             | string    | `Platform::getDefaultPath()` | Per-OS default, see `platform.cpp`. |
| `WindowPosAndSize`     | bytearray | unset               | Opaque Qt `saveGeometry()` blob ŌĆö don't try to reuse, re-derive from screen metrics. |
| `ThemeColor`           | string    | `Theme::DEFAULT_THEME_COLOR` | Hex `#rrggbb`. |
| `AutoMode`             | bool      | `true`              | Follow OS dark/light. |
| `DarkMode`             | bool      | `false`             | Only when `AutoMode=false`. |
| `R5/ShowTermsOnStart`  | bool      | `true`              | First-run ToS modal. |
| `BuddyName`            | string    | `""`                | Empty ŌćÆ fall back to OS username. |
| `Notification`         | bool      | `false`             | Notify on receive. |
| `CloseToTray`          | bool      | `false`             |       |

**How the Wails port handles migration.** `settings.OpenWithMigration` is called at startup. On first run it reads the Qt store (registry / plist / INI, depending on OS) via the matching Go shim, maps the keys above onto the Wails `Values` struct, and writes a fresh JSON file under `<os.UserConfigDir()>/dukto/settings.json` (i.e. `%AppData%\Dukto\` on Windows, `~/.config/dukto/` on Linux, `~/Library/Application Support/Dukto/` on macOS). `WindowPosAndSize` is skipped ŌĆö its format is Qt-internal. After the seed, the Qt store is left untouched so users can downgrade to Qt without losing their Qt settings (at the cost of a one-way fork for any later changes).

Security settings (whitelist, block list, audit-log enable, rate-limit cooldowns, session caps, etc.) have no Qt counterpart and are written straight into the new JSON store with conservative defaults (whitelist off, confirm-unknown off, audit on).

Avatar file (`avatar.png` under `QStandardPaths::AppLocalDataLocation`) can be copied as-is to the Wails data dir.

## Test strategy (applies to both stacks)

Fixtures are generated from the Qt code itself (see `PROTOCOL.md` ┬¦7) ŌĆö no Wireshark needed. Sequence:

1. **Refactor `Sender` to write to a `QIODevice *`** instead of an owned `QTcpSocket`. This makes it unit-testable and decouples the serialisation from the transport. Low-risk, localised change.
2. **Write a Qt fixture generator** (`tests/fixture_gen/`) that drives the refactored `Sender` and `BuddyMessage` against fixed inputs and writes `.bin` + `.json` files under `tests/fixtures/`.
3. **Encoder tests** (Qt and Go): serialise the same inputs, assert byte-for-byte equality with fixtures. The Go side already has its own round-trip and invalid-input tests under `wails/internal/protocol/*_test.go` that mirror the Qt `BuddyMessage::parse` rules.
4. **Decoder tests** (Qt and Go): feed fixture bytes into the receiver pipeline, assert reconstructed filesystem/text matches the `.json` manifest.
5. **Cross-stack integration** (manual, then CI): Qt sender ŌåÆ Go receiver, Go sender ŌåÆ Qt receiver, across all fixture scenarios.
6. **Fuzz** the Go decoder for the TCP stream (hostile peer or corrupted bytes) ŌĆö the Qt code has been in production long enough to be resilient; the Go port won't be on day one.

## Decided stack

- **Wails v2** ŌĆö stable, production-proven. v3 is still beta.
- **Backend**: Go. File I/O + net/ip maps one-for-one to the Qt code; no third-party networking lib needed (stdlib `net` is enough).
- **Frontend**: Svelte. QML ŌåÆ Svelte mapping is roughly per-page (`BuddiesPage.qml` ŌåÆ `Buddies.svelte`, etc.).
- **IPC**: Wails bindings (Go methods callable from JS). The surface is split across `bindings_*.go` files but all hang off the same `App` struct ŌĆö the mental model from the Qt `GuiBehind` still applies.

## Open questions (not blocking)

- **Single-instance on Windows**: not Wails-provided. Options: `github.com/juju/mutex` (cross-platform), named mutex via `golang.org/x/sys/windows`, or a TCP socket on a fixed local port. Pick once we get to the polish phase.
- **Screen capture**: Qt uses `QScreen::grabWindow`. Go equivalents: `github.com/kbinani/screenshot` (pure Go, cross-platform).
- **Clipboard**: `github.com/atotto/clipboard` or Wails' built-in clipboard API.
- **Notifications**: `github.com/gen2brain/beeep` is the usual pick, OS-native under the hood.
